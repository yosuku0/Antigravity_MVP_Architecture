# Antigravity Phase G Requirements: Scaling & Self-Evolution

## 1. 目的
Phase F までに完成した「堅牢な実行基盤」をベースに、運用のスケーラビリティ（量）と、ナレッジの質的な進化（質）を両立させる機能を実装します。

## 2. 主要機能要件

### [G1] インタラクティブ HITL (Slack Integration)
ターミナルに縛られず、モバイルや Slack 経由でジョブを監視・承認・拒否できる環境を構築します。

- **技術設計アプローチ**:
    - **Slack Bolt (Python) の導入**: `apps/daemon/slack_adapter.py` を新設し、Gate 2 の承認・却下が必要なジョブを Slack チャンネルに通知する。Slack は Gate 2 の通知・承認・却下のみを扱い、Gate 3 は CLI のみで実行する。
    - **インタラクティブボタン**: Slack 上の Approve/Reject ボタンから `scripts/approve.py` 相当の API/ロジックを非同期に呼び出し。
    - **非同期状態管理**: Slack からの入力を JOB ファイルに反映し、`wiki_daemon` が次回のスキャンでそれを検知して処理を再開。

### [G2] 並列ジョブスケジューリング & リソース管理
多数のジョブを効率的に処理するための並列実行エンジンを構築します。

- **技術設計アプローチ**:
    - **ジョブキュー・マネージャー**: `wiki_daemon` を拡張し、`RUNNABLE` なジョブを複数のワーカープロセス（またはスレッド）に割り当て。
    - **Docker リソース制限**: 各ジョブを実行するコンテナに CPU/メモリの制限（例: `--cpus 1 --memory 512m`）を動的に設定し、ホストのリソース枯渇を防止。
    - **分散ロック (Locking)**: 複数のワーカーが同一のジョブや出力先を同時に編集しないよう、`work/locks` 下に原子的なディレクトリ/ファイルベースのロック機構を実装。

### [G3] 継続的ナレッジ合成 (Continuous Synthesis)
拒否理由や実行エラーを「筋肉」として蓄積し、AI の精度を自律的に向上させます。

- **技術設計アプローチ**:
    - **フィードバック・インデクサー**: `scripts/approve.py` で記録された拒否理由（Markdown ボディ）を定期的にスキャンし、ベクトル化して `Knowledge Vector DB` (ChromaDB/FAISS) に保存。
    - **コンテキスト注入**: `plan_executor` 起動時に、現在のタスクに類似した「過去の拒否事例」を RAG で検索。
    - **プロンプト動的構築**: 「過去、同様のタスクで『コメント不足』と指摘されました。今回は詳細なドキュメントを含めてください」といった「経験に基づいた制約」をプランに自動挿入。

## 3. 成功指標
1.  **ゼロ・ターミナル運用**: Gate 2 の通知・承認・却下を Slack で完遂できること。Gate 3 の最終昇格承認は CLI のみとする。
Slack は canonical wiki への書き込み権限を持たず、wiki 書き込みは `promote.py --mode execute` のみで行う。
2.  **スループット 3x**: 同時実行数の増加により、ジョブ処理速度が 3 倍以上に向上すること。
3.  **再拒否率の低下**: 過去に指摘された内容と同一の理由で Reject される回数が有意に減少すること。
