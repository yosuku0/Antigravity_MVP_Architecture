# ADR-001: MVP Control Plane — Orchestration Simplification

## Status: Accepted
## Date: 2026-04-24
## Author: Chief MVP Architect
## Context: NIM-Kinetic Meta-Agent MVP

---

## 1. Problem Statement

The integrated design references LangGraph, CrewAI, AI-Q, and potentially ChatDev as orchestration or execution mechanisms. For an MVP, multiple orchestration primitives create overlapping control planes, ambiguous retry semantics, and debugging complexity that directly threatens delivery. We must pick **exactly one** primary orchestrator and constrain all others to subordinate roles.

---

## 2. Decision: LangGraph is the Sole Primary Orchestrator

### 2.1 Why LangGraph (and not CrewAI or AI-Q as primary)

| Criterion | LangGraph | CrewAI | AI-Q |
|---|---|---|---|
| State machine with explicit transitions | Native (`StateGraph`) | No — task-level only | Partial (LangGraph-based) |
| Human-in-the-loop interrupts | Native (`interrupt_before` / `interrupt_after`) | No | Blueprint-dependent |
| Checkpoint / resume after crash | Native (`SqliteSaver`) | No | Blueprint-dependent |
| Retry and fallback policy per node | Native (`RetryPolicy`) | No | No |
| Observability (state at every step) | Native (state inspection) | Limited | Limited |
| Python ecosystem maturity | Confirmed stable | Stable but narrower | Blueprint, not product |

LangGraph is the only option that provides **state machine + checkpoint + HITL interrupt** as first-class primitives. CrewAI provides role-based agent execution; AI-Q provides research patterns. Both are valuable but must operate **inside** LangGraph nodes, not alongside LangGraph.

### 2.2 What this means concretely

- `graph.py` owns the job lifecycle.
- Every state transition (CREATED → CLAIMED → EXECUTING → ...) is a LangGraph node or edge condition.
- CrewAI runs only inside the `crew_execute_node`.
- AI-Q patterns run only inside the `research_node` (which may be a simplified NIM call for MVP).
- No CrewAI crew may spawn another crew. No AI-Q node may trigger a new LangGraph subgraph.

---

## 3. Decision: CrewAI Remains in MVP with Strict Boundaries

### 3.1 What CrewAI owns

- **Single responsibility:** Execute the implementation plan generated by the planning node.
- **Single crew per job:** One `Crew` instance per `crew_execute_node` invocation.
- **Static roles only:** Roles (Brain, Developer, Reviewer) are loaded from YAML config at node start. No runtime role generation.
- **No state machine ownership:** CrewAI reports success/failure/artifact back to LangGraph. CrewAI does not transition JOB status.

### 3.2 What CrewAI does NOT own

- JOB lifecycle management
- HITL gate logic
- Router/provider selection
- Retry policy
- Checkpointing

### 3.3 Rationale for keeping CrewAI

CrewAI role definitions map well to the Brain/Developer separation already designed. Removing it would require rebuilding role-based task decomposition inside LangGraph nodes — reimplementation without added value. Constraining it avoids the complexity while keeping the value.

---

## 4. Decision: Nested Orchestration is Forbidden

### 4.1 Maximum execution depth: 1

```
LangGraph graph.py (depth 0)
  └─ crew_execute_node (depth 1)
        └─ CrewAI crew.run() (depth 1 — terminal)
```

No node may invoke another LangGraph graph. No crew may spawn sub-crews. No job may create another job. Depth > 1 is a runtime error that transitions the job to FAILED.

### 4.2 Why depth 1

Nested orchestration creates:
- Unclear ownership of checkpointing (which graph resumes?)
- Unclear retry boundaries (retry the crew? the node? the whole graph?)
- State explosion in SQLite
- Debugging hell

MVP must prove the single-layer loop before adding any recursion.

---

## 5. Decision: Retry and Fallback Budgets are Hard-Capped

### 5.1 Per-node retry policy

```python
from langgraph.pregel import RetryPolicy

NODE_RETRY_POLICY = RetryPolicy(
    max_attempts=3,
    retry_on=(TimeoutError, ConnectionError, HTTPStatusError),
    # Do NOT retry on ValueError, KeyError, or assertion failures
)
```

### 5.2 Per-job provider-switch budget

```python
MAX_PROVIDER_SWITCHES = 2  # primary → fallback_1 → fallback_2
```

After 2 provider switches, the job transitions to FAILED with `failure_reason: provider_budget_exhausted`. No infinite fallback chains.

### 5.3 Per-job total retry budget

```python
MAX_TOTAL_RETRIES = 5  # across all nodes in one job execution
```

Prevents jobs from looping indefinitely due to transient errors.

### 5.4 Fallback is allowed only on technical failure

Allowed triggers:
- HTTP timeout (> 30s)
- HTTP 5xx
- Connection reset
- DNS resolution failure

Forbidden triggers:
- "The model output was low quality" (subjective — route to audit instead)
- "The model refused the request" (content policy — human review)
- "The output was too short" (specification error — fix prompt, not provider)

## 6. Decision: Explicit Forbidden List for MVP

These capabilities are **prohibited** in MVP to reduce debugging surface area:

| # | Forbidden Capability | Why Forbidden | Detection |
|---|---|---|---|
| 1 | **Recursive job spawning** | A JOB must not create another JOB file. Prevents infinite chains. | Static: scan `crew_execute_node` output for `JOB-` pattern; dynamic: watchdog ignores jobs created by daemon user |
| 2 | **Dynamic crew/agent creation** | All agents/crews are statically defined in config. No runtime `Agent()` instantiation. | Code review + import audit |
| 3 | **Self-modifying router config** | `routing_config.yaml` is read-only at runtime. No code writes to it. | Filesystem read-only flag or explicit block list |
| 4 | **Auto-promotion to wiki** | `promote.py --mode execute` may run only after HITL Gate 3 approval has been recorded in JOB frontmatter. No LangGraph node, Slack action, or autonomous agent may write canonical wiki content. | `promote.py` validates Gate 2/3 approval metadata before execute; Graph has no promote node; Slack is Gate 2 only. |
| 5 | **Auto-merge to main** | Git merge is human-only. Agent may generate diff, never apply it. | `audit.py` checks for `git merge` / `git push` in artifacts |
| 6 | **Dynamic model_policy loading** | Model selection uses static `routing_config.yaml` only. No per-request model overrides. | Router rejects `model_override` parameter |
| 7 | **Nested LangGraph subgraphs** | No `subgraph.invoke()` inside nodes. One graph per job. | Static analysis of node imports |
| 8 | **Autonomous messaging** | Agents must not send messages to Slack/Discord autonomously. | Slack integration is allowed ONLY through `apps/daemon/slack_adapter.py` for Gate 2 HITL notification, approval, and rejection. Slack is NOT allowed for Gate 3. Slack cannot write canonical wiki content. |

## 7. Consequences

### Positive

- Single control plane means single checkpoint, single retry policy, single observability stream.
- Hard numeric caps prevent runaway jobs.
- Forbidden list gives code reviewers explicit red lines.
- Simplifies verification: if it compiles and doesn't use forbidden imports, it's structurally valid.

### Negative

- Complex multi-agent workflows (e.g., research + code + review in parallel) must be sequentialized into LangGraph node chains. This adds latency.
- No self-healing (retry budget exhaustion → human intervention).
- No dynamic load balancing across providers.
- No automatic checkpoint resume in MVP. A daemon crash during execution results in `FAILED` state; the operator must clone to a new JOB. Checkpoint resume is P1.

### Mitigations

- Parallel execution is deferred to Phase 1. Sequential correctness is prerequisite.
- Self-healing is deferred to Phase 2 (Data Flywheel).
- Load balancing is irrelevant for MVP (single user, single job stream).

---

## 8. Migration Notes

If future phases relax these constraints, each relaxation requires:
1. A new ADR.
2. Updated `verification_plan.md` tests.
3. Updated `job_lifecycle_spec.md` state machine.
4. HITL gate position re-evaluation.

No informal relaxation is permitted.
